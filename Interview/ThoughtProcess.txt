                         THE DEVELOPER MINDSET
===================================================================================

Before we dive into code, understand this:

    CODING IS NOT ABOUT WRITING CODE.
    CODING IS ABOUT SOLVING PROBLEMS.

The code is just the final step. 90% of development is:
    - Understanding the problem
    - Planning the solution
    - Breaking it into small pieces
    - Researching how to implement each piece

If you can think clearly, you can code anything.

    const mongoose = require("mongoose");
    
    // This is like defining a table structure
    const listingSchema = new mongoose.Schema({

            type: String,
            required: true    // Can't be empty
        },
        description: String,   // Optional
        image: {
            type: String,
            default: "https://via.placeholder.com/300"  // Default image
        },
        price: Number,
        location: String,
        country: String
    });
    
    // Create the model (like a class)
    const Listing = mongoose.model("Listing", listingSchema);
    
    // Export so other files can use it
    module.exports = Listing;

                     PHASE 7: ADDING LANGGRAPH (FOR AI WORKFLOWS)
===================================================================================

THOUGHT PROCESS:
"My AI needs multiple steps. LangGraph is perfect for this!"

WHAT IS LANGGRAPH?
Think of it as a flowchart for AI:
    
    Step 1 â†’ Step 2 â†’ Step 3 â†’ Result
    
But the AI decides what to do at each step!

BUILDING A LANGGRAPH WORKFLOW:

STEP 1: Define State (What data flows through?)

    from typing import TypedDict
    
    class TravelState(TypedDict):
        destination: str
        research: str
        hotels: str
        activities: str
        final_plan: str

STEP 2: Create Nodes (What does each step do?)

    def research_destination(state):
        # Call AI to research
        result = llm.invoke(f"Research {state['destination']}")
        return {"research": result.content}
    
    def find_hotels(state):
        # Call AI to find hotels
        result = llm.invoke(f"Find hotels in {state['destination']}")
        return {"hotels": result.content}
    
    def plan_activities(state):
        # Call AI to plan activities
        result = llm.invoke(f"Plan activities in {state['destination']}")
        return {"activities": result.content}

STEP 3: Connect Nodes (Build the flowchart)

    from langgraph.graph import StateGraph
    
    # Create graph
    graph = StateGraph(TravelState)
    
    # Add nodes
    graph.add_node("research", research_destination)
    graph.add_node("hotels", find_hotels)
    graph.add_node("activities", plan_activities)
    
    # Connect them
    graph.set_entry_point("research")
    graph.add_edge("research", "hotels")
    graph.add_edge("hotels", "activities")
    graph.set_finish_point("activities")
    
    # Compile
    workflow = graph.compile()

STEP 4: Run It!

    result = workflow.invoke({"destination": "Goa"})
    print(result["activities"])

                     PHASE 8: DEBUGGING MINDSET
===================================================================================

Bugs are NORMAL. Even senior developers spend 50% time debugging.

THE DEBUGGING PROCESS:

    1. DON'T PANIC
       Take a breath. Read the error calmly.
    
    2. READ THE ERROR MESSAGE
       It usually tells you exactly what's wrong!
       
       Example: "Cannot read property 'title' of undefined"
       Translation: You're trying to access .title on something that doesn't exist
    
    3. FIND THE LINE
       Errors show line numbers. Go there.
    
    4. TRACE BACKWARDS
       "Where did this value come from?"
       "Was it passed correctly?"
       "Is the database returning data?"
    
    5. USE CONSOLE.LOG
       Add logs everywhere:
       
       console.log("Got here!");
       console.log("Data is:", data);
       console.log("Type is:", typeof data);
    
    6. GOOGLE THE ERROR
       Copy the error message, paste in Google.
       Stack Overflow usually has the answer.
    
    7. RUBBER DUCK DEBUGGING
       Explain your code to someone (or a rubber duck).
       Often you'll find the bug while explaining!
